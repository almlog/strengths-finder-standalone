作業完了の検証と報告を行います。対象: $ARGUMENTS

## 目的
「完了したと報告したが実際は未完了だった」という行き違いを防止する。
報告前に全ての成果物を実際に確認し、完了事項と未完了事項の両方を明示する。

## 過去の失敗事例

1. **ビルド成功を報告したが中身を確認していなかった（2026-02-13）**
   - CIビルド `Compiled successfully.` を見て即報告
   - ビルド成果物に変更内容が含まれているか未検証
   - 実際には含まれていたが、検証せずに報告するプロセスが問題

2. **未push変更の報告漏れ（2026-02-13）**
   - ローカルでの変更・ビルドのみで「完了」と報告
   - commit/pushが未実施であることを明示しなかった
   - ユーザーが本番環境を確認して初めて未反映に気づく

## 検証ステップ（報告前に全項目必須）

### Step 1: 変更成果物の実確認
- [ ] 変更したファイルが実際に期待通りの内容になっているか `Read` / `Grep` で確認
- [ ] UIの変更は開発サーバー（http://localhost:3006）で目視確認済みか
- [ ] ビルド成果物（`build/`）に変更が含まれているか検証（日本語はUnicodeエスケープされるため、固有の英数字キーワードで検索）

### Step 2: Git状態の確認
- [ ] `git status` で未コミット変更を確認
- [ ] `git diff --stat` で変更量を確認
- [ ] 意図しないファイルが含まれていないか（.env, credentials等）

### Step 3: パイプライン状態の確認
- [ ] commit済みか / 未commitか
- [ ] push済みか / 未pushか
- [ ] CI/CDが実行中か / 成功か / 失敗か
- [ ] Cloud Function変更がある場合、デプロイ済みか

## 報告フォーマット（必須）

```
## 完了報告

### 完了事項
- [実際に確認した完了事項を列挙]

### 検証結果
- ビルド成果物: [検証方法と結果]
- 開発サーバー: [確認結果]

### 未完了・残作業
- [未完了事項を全て列挙。なければ「なし」]

### デプロイ状態
- commit: [済/未]
- push: [済/未]
- CI/CD: [実行中/成功/失敗/未実行]
- Cloud Function: [デプロイ済/未デプロイ/該当なし]
- 本番反映: [反映済/CI通過待ち/未push]
```

## 絶対禁止事項
- 「ビルド成功」だけを報告して中身を検証しない
- 未完了事項を省略して完了だけ報告する
- commit/push状態を報告せずに「完了」と言う
- 「たぶん含まれている」で済ませる。実データで確認する
